## 2026-05-17

integer to roman. used the trick of putting the six subtractive pairs (CM, CD, XC, XL, IX, IV) directly into the value table so the greedy loop never has to undo anything. way cleaner than special-casing them after the fact. always forget which letters are subtractive vs additive until i write it out.

## 2026-05-19

divide two integers without using mul/div/mod. doubled the divisor with left shifts and subtracted each time it fit - basically binary long division. the only annoying part is the INT_MIN / -1 overflow case, returns INT_MAX. didn't bother negating INT_MIN in two's complement land since python ints are unbounded; just worked with abs values and tracked sign separately.

## 2026-05-20

combination sum ii. same shape as 39 but each candidate is single-use and the input can have dupes, so the combinations have to be deduped. sort the array, then in the loop skip when i > start and candidates[i] == candidates[i-1] - that drops duplicate values at the same recursion level but still lets the same value reappear deeper in the tree. also break when the current value exceeds the remaining target since we are sorted, saves a chunk of the recursion.

## 2026-05-28

number of 1 bits. went with the brian kernighan trick - n &= n - 1 clears the lowest set bit, so the loop only runs once per 1-bit. the obvious 32-iteration shift-and-mask works too but felt lazy. light day, just one problem.

## 2026-05-29

insert interval, the natural follow-on to 56. since the input is already sorted you don't need to re-sort - just walk it in three phases: copy everything that ends before new starts, absorb the overlapping run into one merged interval (mutating newInterval in place is fine here), then copy the tail. cleaner than building a fresh sorted list and rerunning merge. one and done for friday.

## 2026-06-02

burst day. cleared a chunk of medium gaps - multiply strings, spiral matrix ii, partition list, surrounded regions, plus pascals triangle as a warmup. multiply strings was the most satisfying, that schoolbook layout into an n+m buffer with the carry going into i+j and units into i+j+1 is one of those tricks you forget until you redo it. surrounded regions i finally stopped doing it inward-out - flooding from the border with a sentinel character is so much less error prone than tracking which O groups touch the edge. partition list was nothing, two dummy heads stitched at the end. ran long enough that i also tidied the readme table and did a small refactor on an older one.

## 2026-06-05

find all anagrams in a string. natural follow-on to the sliding window stuff from earlier in the week. the obvious way is to recompute the whole 26-letter window count and compare to p's count every step, but that comparison is O(26) per index. nicer trick: keep a running "matches" counter of how many of the 26 letters currently agree between window and target, and when the window slides only two letters change, so you fix up matches for just those two. window is an anagram exactly when matches == 26. kept a Counter-based variant in solution_alt for readability since the match-counting one reads a little dense.

## 2026-06-07

unique paths ii, the obstacle version of 62. lazy sunday so just one. the trick is realizing you don't need the full 2d table - one rolling row works because each cell only ever looks up (the value already sitting in dp[c] before you overwrite it) and left (dp[c-1]). obstacles just slam the cell to 0 so nothing routes through it. the one gotcha is the start cell: if the grid begins on an obstacle the whole thing is 0, so seed dp[0] conditionally instead of blindly setting it to 1.

## 2026-06-09

subsets ii. the dup-free version (78) is just append-current-then-recurse, but once the array has repeats you can produce the same subset down two different branches. sorting puts equal values adjacent, and then the one rule that fixes everything is: at a given recursion depth, only the first occurrence of a value may be chosen - any later sibling with the same value is skipped via the `i > start` check. tripped me up for a second because the guard is `i > start`, not `i > 0`; you DO want to pick the duplicate when it's the first element of a deeper branch (that's a genuinely new subset), you only skip it when it's a sibling at the same level. one and done for tuesday.

## 2026-06-18

did a little tree + linked list batch today. preorder iterative was the warmup - the only thing to remember is pushing right before left so left pops first. the one that took a beat was remove-duplicates-from-sorted-list-ii (82): it's deceptively different from 83. 83 keeps one copy of each value so you just skip-forward; 82 deletes the value entirely if it ever repeats, so you can't advance prev into a run - you have to walk cur to the end of the run and splice prev.next past the whole thing. dummy head pays off here because runs can start at the head.

## 2026-06-21

heavy sunday, felt like clearing a few. started easy with remove linked list elements (203) to warm up. classic case for a dummy head - the only thing that makes this annoying is when the nodes you want gone are right at the front, because then there's no prev to re-link. stick a sentinel before head and that whole branch disappears: prev sits on the last kept node, cur scans ahead, and you only advance prev when you actually keep something. return dummy.next at the end since the head itself may have been one of the deleted values.

then game of life (289), which i wanted to do without a second board. the rule you have to respect is that all cells flip simultaneously off the *current* grid, so if you write a plain 0/1 you poison the neighbour counts of cells you haven't visited yet. the bit-packing trick is clean: keep the original live/dead in bit 0 and stash the next state in bit 1, only ever reading `cell & 1` while you scan, then a second pass shifts everything right by one so the future becomes the present. constant extra space, two passes. the live/dead/born rules actually collapse nicely - survive on 2 or 3 if currently alive, born on exactly 3 if dead.

## 2026-06-22

majority element ii (229), the n/3 sequel to 169. the jump that makes it click is realizing there can be at most two values over n/3 - a third would push the total past n - so you run two boyer-moore counters side by side instead of one. the cancellation case is the bit i always have to slow down on: an element matching neither candidate doesn't just decrement one counter, it knocks a vote off *both*, same way the n/2 version pairs unequal elements. and unlike 169 the survivors aren't guaranteed, so you need the verify pass at the end counting actual occurrences before you trust them. monday warmup.

## 2026-06-28

quiet sunday, just wanted one. did construct binary tree from inorder and postorder (106), the mirror of the 105 i did a couple weeks back. same divide and conquer skeleton but two things flip. the root is now the *last* element of postorder instead of the first of preorder, and because i'm walking postorder backwards the roots come out as root, right, left - so when i recurse i have to build the right subtree before the left, otherwise the index pointer gets handed to the wrong side and the whole thing scrambles. inorder still does the same job: find the root's position and everything left of it is the left subtree, everything right is the right. hash the inorder indices once so each lookup is O(1) and it's linear overall.

wrote the iterative stack version too just to keep it honest. that one walks postorder from the back and uses inorder as a guide for when to stop going down the right spine and switch to attaching a left child - you pop while the top of the stack matches the current inorder value, and the last node you popped is where the left child hangs. fiddlier than the recursion but no call stack and it made me actually understand why the right-before-left ordering falls out.

## 2026-07-05

sunday but ended up doing more than i meant to. warmed up on excel sheet column number (171) which is just base-26 done left to right, except the sneaky part is there's no zero digit - A is 1 not 0, so AA is 27 and the whole thing is a *bijective* base rather than plain positional. once you see it's `result = result*26 + (ord(ch)-ord('A')+1)` it's the same loop as parsing a decimal string.

then maximum product of word lengths (318). the trick that makes it fast is that the alphabet is only 26 letters, so each word's letter *set* fits in a single int as a bitmask. two words share no letter exactly when `mask_i & mask_j == 0`, one AND instead of a set intersection. precompute all the masks once and the pairwise loop is cheap.

## 2026-07-12

sunday afternoon, sat with max frequency stack (895). the design trap is thinking you need to reorder anything on pop. you don't. keep a freq map for how many copies of each value are live, and a second map from a frequency f to a stack of every value that has *reached* count f. when i push a value the second, third time it climbs into the group for 2, then 3, and so on, so each group ends up holding its members in push order for free. pop just grabs the top of the group at the current maxfreq - that value is simultaneously the most frequent and, among ties, the newest, which is exactly the two rules the problem wants. drop maxfreq by one when its group empties. O(1) both ways, no heap.

wrote the heap alt too since that's where my head went first: push (-freq, -seq, val) so a min-heap pops highest freq and newest-within-freq. correct but O(log n) per op and it carries a monotonic seq counter, so it's really just the grouped-stacks idea paying interest. kept it for contrast.

## 2026-07-13

monday, came back to reverse pairs (493) after doing count of smaller numbers after self (315) a few days back, since they're the same divide-and-conquer skeleton and i wanted to see where they diverge. both ride merge sort and count while the two halves are sorted, but the difference is the predicate. in 315 the "how many smaller sit to the right" question lines up with the merge comparison itself, so you can fold the counting straight into the merge loop. here the pair condition is nums[i] > 2*nums[j], and that factor of 2 doesn't agree with the plain merge order - a left value can beat 2*right yet still be less than right in the raw comparison - so trying to piggyback on the merge gave me wrong counts until i pulled the counting out into its own pass.

the clean version does a separate two-pointer sweep before merging: for each value in the sorted left half, push j forward across the right half while left > 2*right, and add j-mid. the key realization is j never resets - because the left half is sorted ascending, a bigger left value only ever needs j to move further right, so the whole sweep is O(n) per level and the total stays O(n log n). after counting i just let python sort the slice back for the merge, which keeps the code short.

wrote the fenwick alt too since 315 has one and i like having the counting-structure version side by side. compress the values and their doubled thresholds into one coordinate space, then sweep left to right asking how many already-seen values exceed 2*nums[i] before inserting nums[i]. the annoying part is the threshold and the value live in the same compressed axis, so i bisect on 2*v against the coord list rather than trusting the rank map for it. same O(n log n), just trades the recursion for a bit-indexed tree.

## 2026-07-17

friday, knocked out 4sum ii (454). the naive read is four nested loops over the arrays, O(n^4), which is hopeless once n gets past a few hundred. the fix is meet in the middle: it's really a two-sum in disguise once you group the four arrays into two pairs. count every a+b from the first two arrays into a Counter, then for each c+d from the last two ask how many stored sums equal -(c+d). both passes are O(n^2) so the whole thing lands at O(n^2) time and space. worth noting this is 4sum ii, not 4sum - because we're returning a *count* of index tuples and the arrays are separate, duplicates aren't a headache the way they are in the classic 4sum, so no sorting or dedup dance here, just the two counters meeting in the middle.

## 2026-07-21

monday, did integer break (343). started with the DP because it's the obvious safe answer - dp[i] is the best product for i, and for each i you try every first cut j and pair it with either the whole remainder or its own best break, max(i-j, dp[i-j]). that inner max is the whole trick: it lets a "part" quietly be a further-broken chunk instead of a literal number. O(n^2) and done.

but the pattern in the dp table nagged at me - the splits kept landing on 3s. worked out why: for any part k >= 5 you're better off peeling a 3 off it because 3*(k-3) > k there, and 4 just ties with 2+2, so an optimal split never keeps a part bigger than 4 and prefers 3 over 2. so the closed form is just: cram in as many 3s as fit, then patch the remainder - remainder 2 leaves a plain 2, remainder 1 is the sneaky one because 3+1 wants to become 2+2, so you trade one 3 for a 4. n=2 and n=3 stay special since "at least two parts" forces a 1 into the split and caps them at 1 and 2. dropped it to O(log n) for the power. kept the dp as the alt since it's the one i'd actually write first in an interview before reaching for the math.

## 2026-07-23

trees day. wanted to nail down the BST mutation trio - insert, delete, and the greater-tree transform - since i always half-remember delete under pressure. started with insert into a bst (701) as the warmup because it's the gentle one: there's no rebalancing and any valid leaf position counts, so you just ride the invariant down (left when smaller, right otherwise) until you fall off the tree, and that empty child is exactly where the value preserves ordering. wrote it recursively first since the "root.left = insert(root.left, val)" reassignment pattern is the same skeleton i'll want for delete, then did the iterative walk as the alt - same path, just carrying a cursor and attaching to the first empty child instead of unwinding a stack. O(h) either way.

then the one i actually came for: delete node in a bst (450). the no-child and one-child cases are easy - return the surviving child and let the parent's "root.left = delete(...)" reassignment splice it out. the two-child case is the whole problem. you can't just yank the node, so you find its in-order successor (leftmost of the right subtree, the next-larger key), copy that value up into the node, then recursively delete the successor from the right subtree. the successor has no left child by definition, so that second delete always lands in an easy case and the recursion can't spiral. tempting to hand-splice the successor instead but routing it back through deleteNode keeps the pointer surgery in one place.

capped the day with vertical order traversal (987), which is really a bookkeeping-and-sorting problem wearing a tree costume. tag every node with (row, col) - root (0,0), left is (row+1, col-1), right (row+1, col+1) - bucket by column, done in one dfs. the trap is the tie-break: unlike plain vertical order, two nodes can share the exact same (row, col), and the spec wants those ordered by value. so it's not enough to keep a bfs order per column; you have to sort each column by (row, value) so the row drives top-to-bottom and value settles same-cell ties. once i sorted on the pair instead of trying to be clever with insertion order it fell out clean. nice bookend to the mutation problems - same trees, totally different muscle.
